Core Concepts

Understand the fundamental building blocks of the Morse protocol the data model, authorization system, storage strategy, and how on-chain and off-chain components interact.

The Data Model

Morse organizes content in a four-level hierarchy. Each level is stored on-chain and has a clear responsibility:

Level 1
📰 Publication
→
e.g. my-blog a brand-level content domain
↓
Level 2
🗂 Collection
→
e.g. articles, site-pages content lanes
↓
Level 3
📝 Entry
→
e.g. welcome-post a single content unit
↓
Level 4
📜 Revision
→
immutable snapshot blob reference + metadata

Editors operate mostly at the Entry / Revision level. Owners operate mostly at the Publication / Authorization level.

Publications

A Publication is the top-level shared object in Morse. It is the permission management boundary for all content under a brand or domain.

  • Identified by an immutable slug (e.g. my-blog) registered in the global PublicationRegistry.
  • Slug uniqueness is enforced globally no two active publications can share a slug.
  • Contains a map of named collections.
  • Tracks a denylist of revoked publisher caps.

Publications use key ability only (no store) sharing and transfer are gated through module functions, not arbitrary object movement.

Collections

A Collection is a named content lane inside a publication. Its storage mode is immutable after creation and determines how content references are stored:

ModeValueBest forBlob reference type
Blob0Growing collections articles, posts, media libraries. Content changes independently.Blob(ID) Sui object ID of a standalone Walrus Blob
Quilt1Small, stable collections site pages, config-like structured documents.QuiltPatch(vector<u8>) 37-byte patch identifier within a shared quilt

Entries inside a collection are keyed by a monotonic entry_id (u64). IDs are stable they never get renumbered after deletions (gaps are expected).

Entries & Revisions

An Entry is a single content unit in a collection. It holds an immutable, append-only revision vector.

The Dual-Head Model

Each entry maintains two head pointers:

  • draft_head points to the latest draft revision. Editors can iterate here without affecting what readers see.
  • public_head points to the currently published revision. Readers and indexers use this.

This dual-head model enables real editorial workflows: drafts can evolve privately while production clients serve stable published content.

What a Revision Contains

Each EntryRevision is immutable once appended. It stores:

  • blob_ref either a Blob(ID) or QuiltPatch(bytes) pointing to the Walrus payload.
  • content_type MIME type string (e.g. application/json, text/markdown).
  • encrypted whether the Walrus payload is Seal-encrypted.
  • access_policy 0 (public), 1 (publisher), or 2 (subscription).
  • seal_id Seal identity for encrypted revisions (required when encrypted = true).
  • author address of the transaction sender who created this revision.

Capability-Based Authorization

Morse uses two capabilities to separate publication permission management from day-to-day publishing:

👑
OwnerCap
One per publication. Required to issue or revoke PublisherCaps, delete the publication, and transfer ownership. Transferable by design ownership can be sold or delegated.
✍️
PublisherCap
Many per publication. Required for all write operations. Bound to a specific holder address a moved or shared cap object cannot be used by unapproved addresses.

Revocation Denylist

Revocation is denylist-based, not allowlist-based. This is an important design choice:

  • Issuing a cap has zero table writes caps are valid by construction.
  • Only revocations write to the revoked_publisher_caps table.
  • This optimizes for the common case (no revocations) and makes issuing caps cheap.
  • Safe because PublisherCap has no store ability and no external constructor caps cannot be fabricated outside the module.
💡
Revocation is immediate
Once a PublisherCap ID is added to the denylist, all future writes using that cap will be rejected without modifying historical content state.

Walrus Storage

Walrus is a decentralized blob storage protocol. Morse uses it to store the actual content payloads and assets, keeping large bytes off-chain while preserving on-chain verifiable references.

  • Walrus is format-agnostic it stores binary blobs. Morse can support JSON, Markdown, images, video, or any custom schema.
  • Contracts store references to blobs (blob_id or QuiltPatchId), not the payload bytes themselves.
  • Blobs must be deletable Morse enforces this by requiring the actual &Blob object (not just an ID) in production write paths.

QuiltPatchId Format

For Quilt mode collections, each revision stores a 37-byte compact identifier:

QuiltPatchId = quilt_blob_id (32 bytes)
             + version     ( 1 byte )
             + start_index ( 2 bytes)
             + end_index   ( 2 bytes)

Access Control & Seal

Morse supports three access policies on content revisions:

PolicyConstantDescription
Public0Unencrypted. Anyone can read the Walrus payload directly.
Publisher1Encrypted via Seal. Only active publishers of this publication can decrypt.
Subscription2Reserved. Planned for subscription-gated content with Seal policies.

Seal is MystenLabs' access control system for encrypted Walrus content. The seal_approve_publisher entry function in the contracts validates that a caller holds an active, non-revoked PublisherCap before approving decryption.

Off-Chain Services

The blockchain is the canonical source of truth, but off-chain services improve the developer experience:

  • Indexer listens to on-chain events and builds a read-optimized state database.
  • Content API serves fast queries for listings, slug resolution, and payload references.
  • SDK TypeScript package that wraps Sui/Walrus/Seal interactions for app developers.
  • CLI command-line tool for scripted automation, CI, and agent-native workflows.
🔌
Open protocol guarantee
Off-chain services are replaceable. Any team can run their own indexer, API, or integrate directly with the contracts via the SDK. Protocol access is never gated by a single service provider.
