Collections

A collection is a named content lane inside a publication. It groups related entries together and determines how their Walrus payload references are stored.

What is a Collection?

Collections organize entries into predictable buckets for editorial workflows and API consumers. A publication holds collections in a VecMap<String, Collection> keyed by name. The choice of VecMap reflects the expectation that a publication will have only a small number of named collections.

Move 
public struct Collection has store {
  name: String,
  storage_mode: u8,        // 0 = blob, 1 = quilt  immutable after creation
  next_entry_id: u64,      // monotonic counter
  entries: Table<u64, Entry>,
}

Entries inside a collection are keyed by a stable monotonic entry_id (u64). IDs are never reused deletions leave holes in the sequence. This ensures stable external references to individual entries.

Storage Modes

Every collection has an immutable storage_mode chosen at creation. This controls how entry revisions reference Walrus content.

mode 0
Blob Mode

Each entry revision references a standalone Walrus Blob object by its Sui object ID. Clean, independent blobs per revision.

mode 1
Quilt Mode

Each entry revision stores a 37-byte QuiltPatchId pointing to a patch within a shared Walrus Quilt blob.

Blob Mode (STORAGE_MODE_BLOB = 0)

In blob mode, every revision stores a BlobRef::Blob(ID) the Sui object ID of a standalone Walrus Blob. Each entry revision is fully independent: uploading new content creates a new blob, and the old blobs remain accessible by their IDs.

Best for:

  • Articles, blog posts, documentation pages
  • Media libraries (images, videos, audio)
  • Any collection where entries change independently and frequently
  • Situations where you want maximum operational simplicity

Quilt Mode (STORAGE_MODE_QUILT = 1)

In quilt mode, every revision stores a BlobRef::QuiltPatch(vector<u8>) a 37-byte identifier that locates a patch within a shared Walrus Quilt blob. This is more compact and efficient when updating small parts of a grouped content set.

QuiltPatchId layout 37 bytes 
// Byte layout of a QuiltPatchId:
┌─────────────────────────────────┬─────────┬───────────────┬─────────────┐
│  quilt_blob_id (32 bytes)       │ ver (1) │ start_idx (2) │ end_idx (2) │
└─────────────────────────────────┴─────────┴───────────────┴─────────────┘
  Walrus blob ID of the quilt       version   start index     end index
  (this is the Walrus blob_id,                (u16 LE)        (u16 LE)
   not the Sui object ID)

The contract validates that every QuiltPatchId is exactly 37 bytes. Shorter or longer byte vectors are rejected with EInvalidQuiltPatchId.

Best for:

  • Website navigation sections or config-like structured documents
  • Compact content sets updated in place (e.g. featured banners, site metadata)
  • Stable grouped content with targeted patch updates

Choosing a Storage Mode

ScenarioRecommended mode
Growing blog / article feedBlob (0)
Frequently updated individual documentsBlob (0)
Media / asset libraryBlob (0)
Static site sections (header, footer, nav)Quilt (1)
Config JSONs updated as a groupQuilt (1)
Small, bounded content setsQuilt (1)
Storage mode is immutable. Once a collection is created with a given storage_mode, it cannot be changed. All entries in the collection must use references that match the mode the contract enforces this on every write. Choose carefully before creating a collection.

Creating a Collection

Collections are created through the publication module's create_collection function, which requires a valid PublisherCap. The constructor is package-private external callers always go through the publication API.

Move 
public fun create_collection(
  publication: &mut Publication,
  cap: &PublisherCap,
  name: String,
  storage_mode: u8,    // 0 = blob, 1 = quilt
  ctx: &mut TxContext,
)

Collection names must be unique within a publication. Creating a collection with a name that already exists aborts with ECollectionAlreadyExists.

Example creating two collections 
// Create a blob-mode collection for articles
publication::create_collection(
  &mut publication,
  &publisher_cap,
  "articles".to_string(),
  0, // STORAGE_MODE_BLOB
  ctx,
);

// Create a quilt-mode collection for site sections
publication::create_collection(
  &mut publication,
  &publisher_cap,
  "site-pages".to_string(),
  1, // STORAGE_MODE_QUILT
  ctx,
);

Deleting a Collection

A collection can be removed from a publication using delete_collection. The collection's entries table must be empty all entries must be deleted first.

Move 
public fun delete_collection(
  publication: &mut Publication,
  cap: &PublisherCap,
  name: String,
  ctx: &mut TxContext,
)
Why entries must be removed first. The Table<u64, Entry> that holds entries requires an explicit destroy_empty Sui's type system does not allow dropping a non-empty table. Delete entries individually with delete_entry_from_collection before removing the collection.

Entry IDs

When an entry is added to a collection, it is assigned a stable, monotonically increasing entry_id: u64. The counter starts at 0 and increments with each insertion.

  • Entry IDs are never reused after deletion
  • IDs are stable external systems can safely hold references by ID
  • Gaps in the ID sequence are expected and normal after deletions
Example ID sequence 
// Add entries: IDs assigned sequentially
entry_id = add_entry_to_collection(...) // → 0
entry_id = add_entry_to_collection(...) // → 1
entry_id = add_entry_to_collection(...) // → 2

// Delete entry 1
delete_entry_from_collection(..., entry_id: 1, ...)

// Next add gets ID 3  ID 1 is never reused
entry_id = add_entry_to_collection(...) // → 3
