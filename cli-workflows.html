CLI Workflows

End-to-end recipes for the morse CLI: publish content, read it back, encrypt and decrypt, delegate write access, and script it in CI. These use the active context, so most commands need no object ids. For the full per-command reference, see Commands.

✅
Before you start
Install the CLI and import an account (see Overview). You need a funded Sui testnet address for gas, and testnet WAL for the content steps.

Publish your first entry

Create a profile, import a key, then create a publication and a collection. create selects each new object as the active one, so the steps after need no ids.

bash 
morse config add testnet --network testnet   # profile (becomes default)
morse account import                          # hidden prompts: key + password
morse publication create --name "My Blog" --slug my-blog   # auto-selected
morse collection create posts --mode blob     # auto-selected

echo "hello morse" > post.md
morse entry add hello --file post.md          # uploads to Walrus, prints a link
text 
Uploading 12 bytes to Walrus...
Added entry #0 "hello". (tx: D9Yfjk...)
  view: https://aggregator.walrus-testnet.walrus.space/v1/blobs/0iKeii...

Check the active context any time with morse status; switch to another publication later with morse use <slug> [collection].

Read it back

entry get shows metadata; entry read fetches the public content itself. The view link printed by entry add opens the same bytes in a browser.

bash 
morse entry get 0              # metadata: revisions, heads, blob ref
morse entry read 0             # content to stdout
morse entry read 0 --out got.md

Publish a new revision

Append a new public revision directly, or stage a draft first and publish it.

bash 
echo "hello morse, edited" > post.md
morse revision publish-direct 0 --file post.md

# or: stage a draft, then promote it
morse revision append-draft 0 --file draft.md
morse revision publish-from-draft 0 1 --file final.md

Encrypted content

Encrypt with Seal, upload, and add an encrypted entry; decrypt it back by signing a short-lived SessionKey with the active account.

bash 
echo "members only" > secret.txt
morse entry add-encrypted members --file secret.txt
morse entry decrypt 1 --out recovered.txt
🔐
Encrypted entries are drafts
An encrypted entry is created as a draft (never public on-chain). Retrieve it with entry decrypt plus a PublisherCap, not entry read. The generated sealId is shown in the --json output of add-encrypted.

Delegate write access

Morse has two roles: the owner (OwnerCap holder) governs the publication, and publishers (PublisherCap holders) write content. The owner issues a PublisherCap to a teammate, who then writes with their own key; the owner can revoke it at any time.

bash 
# owner, on the active publication, issues a cap bound to the teammate
morse cap issue 0xTEAMMATE_ADDRESS            # prints the new PublisherCap id

# the teammate writes with their own key; the cap is auto-resolved
MORSE_PRIVATE_KEY=suiprivkey1... morse entry add their-post --file post.md -P my-blog -C posts

# owner revokes; further writes with that cap abort on-chain
morse cap revoke 0xCAP_ID

A publication's name and slug are immutable on-chain (the slug is the registry's unique key), so there is no rename command. List who holds caps with morse cap list [address].

Scripting and CI

Authenticate with env vars (no prompts), skip confirmations with --yes, and parse --json on stdout. Reads never need a key.

bash 
# MORSE_PRIVATE_KEY skips the keystore and password entirely
export MORSE_PRIVATE_KEY=suiprivkey1...
export MORSE_NETWORK=testnet

pub=$(morse --json publication create --name CI --slug "ci-$(date +%s)" | jq -r .publicationId)
echo "created $pub"
morse --yes publication delete "$pub"

# exit codes are scriptable: 0 ok, 2 usage, 3 not found, 4 auth, 5 network
morse publication get 0x00 || echo "exit $?"
