Publishing

The six operations consumers actually call: create a publication, add a collection, upload bytes and attach them as an entry (plain or encrypted), read them back. Every snippet on this page comes from a compile-checked example in morse-sdk/examples/.

Lifecycle

Publications hold collections; collections hold entries; entries hold an append-only vector of revisions. Permission management lives in Sui; payloads live in Walrus.

1. Container
Publication unique slug, OwnerCap + PublisherCap
↓
2. Bucket
Collection named, storageMode fixed at create
↓
3. Item
Entry monotonic u64 id, name, draftHead / publicHead
↓
4. Payload
Revision BlobRef + contentType + access policy + sealId

The examples below assume you've built a context as shown in Client Setup i.e. you have adapter, client, config, and reader in scope. Browser flows also carry a signer built via WalletStandardSigner.fromAccount.

1. Create a publication

Returns the publication's ID along with an OwnerCap (governance) and the first PublisherCap (write authority), both transferred to the adapter's address atomically.

typescript 
import { ContractAbortError, createPublication } from "morse-sdk";

try {
  const created = await createPublication(adapter, config, {
    name: "My Publication",
    slug: "my-publication",
  });
  // created.publicationId, created.ownerCapId, created.publisherCapId
} catch (err) {
  if (err instanceof ContractAbortError && err.reason === "ESlugAlreadyExists") {
    // slug is globally unique - pick another or look up the existing publication
  }
  throw err;
}

2. Add a collection

Collections are named buckets for entries. storageMode is fixed at creation; switching to Quilt for the same name later requires deleting and recreating.

typescript 
import { createCollection, StorageMode } from "morse-sdk";

await createCollection(adapter, config, {
  publicationId: created.publicationId,
  publisherCapId: created.publisherCapId,
  name: "blog",
  storageMode: StorageMode.Blob,  // or StorageMode.Quilt for packed small payloads
});

3. Add an entry (the headline)

Upload bytes to Walrus and attach them as the first revision of a new entry in 2 wallet popups (one for register_blob, one for the combined certify_blob + add_entry_to_collection PTB). Entry IDs are monotonic u64s assigned by the contract; the SDK retrieves them via a pre-flight simulation.

typescript 
import {
  addEntryFromBytes,
  DefaultWalrusWriteAdapter,
} from "morse-sdk";

const walrus = DefaultWalrusWriteAdapter.fromConfig(
  { network: "testnet", suiClient: client },
  keypair,  // or `signer` in the browser
);

const entry = await addEntryFromBytes(adapter, config, {
  walrus,
  publicationId: created.publicationId,
  publisherCapId: created.publisherCapId,
  collectionName: "blog",
  name: "first-post",
  bytes: new TextEncoder().encode("hello world"),
  contentType: "text/plain",
  upload: { epochs: 3, deletable: true },
});
// entry.entryId is the stable monotonic ID (0, 1, 2, ...)
⚡
2 wallet popups, not 3
addEntryFromBytes uses the Walrus flow-aware capability to combine certify_blob and add_entry_to_collection into a single PTB. The HTTP publisher path collapses this further to 1 popup; see Client Setup → Walrus adapters.
⚠️
Handle UncertifiedBlobError
If popup 1 (register + upload) succeeds but popup 2 (certify + add_entry) fails user rejected, contract aborted, network blip the SDK throws UncertifiedBlobError carrying blobObjectId and blobId. The blob is on storage nodes and you've paid for it but it's uncertified; storage releases on registration expiry. Surface the error to your user or log the IDs for support.

4. Encrypted entries

Encrypt with Seal client-side, upload ciphertext to Walrus, attach to an encrypted entry same 2 popups. The contract stores encrypted=true and the sealId bytes per revision; accessPolicy is hardcoded to Publisher (Subscription is reserved for future work).

typescript 
import {
  addEncryptedEntryFromBytes,
  buildPublisherSealId,
  DefaultSealAdapter,
  DefaultWalrusWriteAdapter,
} from "morse-sdk";

const seal = DefaultSealAdapter.fromMorseConfig(config, {}, client);
const walrus = DefaultWalrusWriteAdapter.fromConfig(
  { network: "testnet", suiClient: client },
  keypair,
);

// Fresh per-entry identity. publicId(32) || tag(1=Publisher) || nonce.
const nonce = crypto.getRandomValues(new Uint8Array(16));
const sealId = buildPublisherSealId(created.publicationId, nonce);

const result = await addEncryptedEntryFromBytes(adapter, config, {
  walrus,
  seal,
  publicationId: created.publicationId,
  publisherCapId: created.publisherCapId,
  collectionName: "secret-blog",
  name: "private-doc",
  plaintext,                          // Uint8Array
  contentType: "application/octet-stream",
  sealId,
  upload: { epochs: 3, deletable: true },
});
🔏
Seal identity structure
The Move contract enforces sealId = publicationId(32) || policyTag(1=Publisher) || nonce(≥1). buildPublisherSealId assembles this for you; use a fresh nonce per identity (the same nonce produces the same identity, so reuse is a deduplication signal). The publication-id prefix scopes the identity to a single publication for cross-publication forgery prevention.

Decrypting later

Decryption requires three things: a Seal SessionKey (built via wallet personal-message signature), the sealId bound to the revision, and the PublisherCap ID the key servers verify is active (not revoked).

typescript 
import { SessionKey } from "@mysten/seal";

const sessionKey = await SessionKey.create({
  address: signer.toSuiAddress(),
  packageId: config.originalPackageId ?? config.packageId,
  ttlMin: 10,
  signer,
  suiClient: client,
});

const plaintext = await seal.decrypt(ciphertext, {
  sessionKey,
  sealId,
  publisherCapId,
});
⚠️
No encrypted publish path
publishFromDraft and publishDirect always write non-encrypted public revisions the Move contract hardcodes encrypted=false on the public path. Encrypted content stays as drafts. Use addEncryptedEntryFromBytes / appendEncryptedDraftRevision and gate reads behind seal.decrypt at delivery time.

5. Read it back

Single-object reads, paginated lists, and an async iterator for full-collection scans. The reader is gRPC-backed at v0.1.0.

typescript 
// Single entry by ID.
const fetched = await reader.getEntry(
  created.publicationId,
  "blog",
  entry.entryId,
);
const firstRevision = fetched.revisions[0];
// firstRevision.blobRef is a discriminated union: switch on `.kind`
// for blob (kind: "blob") vs quilt (kind: "quilt") references.

// Paginated list.
const page = await reader.listEntries(
  created.publicationId,
  "blog",
  { pageSize: 50 },
);

// Stream every entry in a collection.
for await (const e of reader.scanEntries(created.publicationId, "blog")) {
  console.log(e.id, e.name);
}
ℹ️
Ordering caveat
listEntries and scanEntries return entries in dynamic-field object-store order, NOT chronological order. Sort by entry.id if you need insertion order.

Choosing the right entry path

The SDK ships two ways to publish content. The high-level addEntryFromBytes (and its encrypted twin addEncryptedEntryFromBytes) is the recommended default; the split form (uploadBlob + addEntry) is for cases the high-level shape doesn't cover.

Use case Function Wallet popups
Publish raw bytes as a new entry (typical case) addEntryFromBytes 2
Publish encrypted bytes as a new entry addEncryptedEntryFromBytes 2
Reuse one blob across many entries (deduplication) uploadBlob once, then addEntry N times 2 + N
Decouple upload and add-entry (draft-then-attach UX) uploadBlob at upload time, addEntry at publish time 2 + 1
Server pre-uploads; browser only adds entries uploadBlob (server), addEntry (browser) 0 server + 1 browser

addEntryFromBytes requires a WalrusWriteAdapter that also implements WalrusFlowCapable (the optimization uses its flow-aware startBlobUpload API). DefaultWalrusWriteAdapter implements both; the HTTP publisher path does not, and the SDK rejects mismatches with TransportError before any I/O.
