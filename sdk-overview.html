SDK Overview

morse-sdk is the TypeScript client for the Morse decentralized CMS protocol. It packages Sui transaction building, Walrus blob upload and download, and Seal threshold encryption behind a small, swappable adapter pattern. Browser and Node, ESM-only, testnet at v0.1.0.

📦
Pre-release testnet only
Move contract addresses are baked in via morseConfig({ network: "testnet" }) and re-pinned on every redeploy. Mainnet support arrives once the contracts are frozen at v1.0.0.

What you get

🧩
Adapter pattern
Four small interfaces wallet, Walrus write, Walrus read, Seal with shipped defaults and easy substitution for custom flows.
🆔
Branded ID types
PublicationId, PublisherCapId, BlobObjectId, WalrusBlobId, QuiltPatchId. Compile-time guards against passing the wrong ID into the wrong call.
🚨
Structured errors
Every throw extends MorseError. Narrow by class + property (e.g. ContractAbortError.reason); no string parsing.
⚡
2-popup uploads
addEntryFromBytes collapses register + certify + add-entry into 2 wallet popups instead of 3 via a flow-aware Walrus capability.
🧪
Testnet smokes
End-to-end scripts/phase-N-*.ts run against live testnet (real WAL, real SUI) to verify the public surface on every release.
✅
97% coverage
265 unit tests, ~97% line / ~96% function coverage at v0.1.0. Lint, typecheck, test, build all gate every change.

Adapter Architecture

The SDK is built around four swappable abstractions. Pick the shipped default or implement the interface against whatever signing / storage / encryption stack you already have.

Wallet
WalletAdapter
KeypairAdapter server / CLI
WalletStandardSigner browser
↕
Walrus write
WalrusWriteAdapter
DefaultWalrusWriteAdapter direct, 2 popups
HttpPublisherWriteAdapter operator, 1 popup
↕
Walrus read
WalrusReadAdapter
DefaultWalrusReadAdapter direct fanout
HttpAggregatorReadAdapter CORS-friendly
↕
Encryption
SealAdapter
DefaultSealAdapter canonical testnet servers

The rest of the SDK domain ops like createPublication, addEntryFromBytes, the RpcPublicationReader composes these adapters. Substituting an adapter never touches the call sites.

Compatibility

morse-sdk is built and tested against specific minor versions of its Mysten substrate. Newer or older versions are not validated and may produce runtime errors. The peer-dependency ranges in package.json enforce these bounds.

Package Version Required
morse-sdk0.1.x(this package)
@mysten/sui^2.16.2Yes types, gRPC client, keypairs
@mysten/walrus^1.1.6Optional for DefaultWalrusWriteAdapter
@mysten/seal^1.1.3Optional for encryption
⚠️
Pin to the verified minor
Mysten ships breaking changes inside major version boundaries. When @mysten/walrus@1.2.0 (or any minor bump) lands, morse-sdk needs a coordinated bump and re-verification. Use bun add morse-sdk@~0.1.0 for patch updates without surprise minors.

Runtime requirements

ESM-only ("type": "module"). CommonJS require is not supported.

RuntimeSupportedNotes
Bun>= 1.2.0Primary development runtime. Smoke scripts use Bun.
Node>= 18.0Uses ES2022 (private fields, Error.cause), TextEncoder, crypto.getRandomValues, BigInt.
BrowserEvergreenChrome / Edge / Firefox / Safari recent stable. Vite, Webpack, esbuild bundlers all handle the SDK with zero polyfills.

The SDK does not pull in Node-specific APIs (fs, path, process, node:crypto); the public surface is portable across runtimes. A handful of @mysten/* substrate libraries reach into Node-shaped APIs internally consult their docs for browser polyfill requirements (typically zero with modern bundlers).

Known limitations

  • Testnet only at v0.x. Mainnet config lands once the contracts are frozen.
  • gRPC client only at v0.1.0. Reader and adapter interfaces are typed against SuiGrpcClient from @mysten/sui/grpc. JSON-RPC fallback is planned for v0.2.0.
  • No encrypted publish path. The Move contract hardcodes encrypted=false on publish_from_draft and publish_direct. Encrypted content stays as drafts.
  • Subscription access policy is reserved, not enforced. Publisher policy is the only enforced encrypted path.
  • listEntries ordering is dynamic-field object-store order, not chronological. Sort by entry.id for insertion order.
  • zkLogin decoder ships, end-to-end unverified. Walrus and Seal SessionKey flows have not been smoke-tested with zkLogin signatures. Use keypair / passkey accounts in production until you've verified your stack.
